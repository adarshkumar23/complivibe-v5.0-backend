## 2. What counts as a "feature", and why the number moved

**This inventory changed methodology.** Prior versions counted **one feature per OpenAPI tag group**, which produced 190. That method is misleading here, because the single tag `ai_governance` covers 379 endpoints spanning sixteen unrelated capabilities, while trivially small tags counted as one feature each.

This version counts a **distinct capability**: a coherent unit of behavior backed by its own tables and/or its own service logic, identified by reading the router and the services it calls. That yields **399**.

**399 and 190 are not comparable, and 399 is not "209 new features."** Nothing was invented; the same
surface is simply resolved at a truthful granularity. Do not diff this number against 158/186/190/201/225.

Feature counts by source of truth:

**How 399 is derived** (reproducible from §5): parse §5 as markdown tables, keep rows with >=4
cells that are neither a header (`| feature |`) nor a `---` separator. That yields 402 feature rows.
Subtract 3 conceptual duplicates — P3 policy-derivation, evidence AI-assist, and Autopilot graph
reasoning each appear both in their home domain and again in the Interconnection section, because
they belong to both stories. Counted once here: **399**.

Parse the tables structurally, not with a slug regex. An earlier revision of this file reported 394
because its regex assumed the first cell was a bare kebab-case slug, silently dropping four real
features whose names carry a parenthetical or a dot — `policy-derivation (P3)`,
`audit-findings-v1 (canonical)`, `pbc-items (canonical)`, and `scim-2.0-provisioning`.

#### Anomalies
##### Cross-cutting mechanisms (confirmed from code)
- **Auth is cookie-first w/ bearer fallback.** BearerOrSessionCookie (OAuth2PasswordBearer subclass, app/core/deps.py:28) accepts Authorization: Bearer **only when APP_ENV != "production"**, else httpOnly cv_session cookie. Cookie-sourced mutating requests require double-submit X-CSRF-Token matching signed csrf JWT claim via secrets.compare_digest. Sessions are real user_sessions rows validated by jti every request.
- **RBAC is a FastAPI dependency**: require_permission("code") → RBACService.user_has_permission, joins Role.is_active so deactivation is immediate. Codes live in app/services/seed_service.py: PERMISSIONS dict line 102 (208 codes), ROLE_PERMISSION_MAP line 313.
- **Rate limiting is Redis-backed but OPT-IN.** RATE_LIMIT_REDIS_URL defaults None (config.py:27); when set → slowapi storage_uri (shared cross-worker) + pub/sub invalidation over per-org TTL cache. **Unset, it silently degrades to per-worker in-memory counting — the N-workers = N×limit bypass returns.**

##### Highest severity
- **Governance signing-key rotation/revocation is COSMETIC.** _derived_signing_key_material (organizations.py:149) computes HMAC(SECRET_KEY, "{org_id}:{purpose}:{key_id}") on demand — no key material ever stored or destroyed. A "revoked" key still produces a valid signature; verify only downgrades a trusted flag. Anyone with SECRET_KEY can forge any org's manifest. **Inconsistent w/ P3 which uses real Ed25519** — platform now has two signing models of very different strength.
- **Custom roles have NO permission ceiling.** _validate_permission_codes only checks each code exists. A holder of org:update can mint a role containing ANY permission and assign it to any membership. No check creator holds what they grant, no self-assignment block — **genuine privilege-escalation path**.
- **SSO/OIDC tokens drifted from the cookie rearchitecture.** SSOService._create_jwt_for_user (sso_service.py:215) mints token w/ org_id/auth_method but **no jti and no csrf claim**, creates no user_sessions row, callbacks never set auth cookies. SSO tokens are unrevocable, invisible to /sessions, and cannot perform mutating requests as a cookie. **SSO login does not produce a usable browser session in production.**

##### Dead / not-wired
- **Nothing drains queued webhook deliveries.** emit() called from 7+ services creating pending rows, but deliver() reachable ONLY via manual POST. Verified: none of the 33 scheduler jobs touches webhooks.
- **Nothing schedules automation rules.** run_due_scheduled_rules exists only in service + own router; next_run_at advances only when a human POSTs /automation/schedules/run-due.
- **Compliance-bot has ZERO outbound delivery.** Digests/alerts written status="pending", nothing drains them. Signature verification is real HMAC but uses custom X-ComplianceBot-Signature, **not Slack's v0: scheme** — a real Slack app cannot call it unmodified.
- **Three export types allowlisted but unrunnable**: compliance_report_pdf/docx/xbrl in ALLOWED_EXPORT_TYPES (export_service.py:44) but _build_data_for_type raises "Unsupported export_type" for all three — job can be created and /run always fails.
- GET /compliance/contracts static literal, identical every tenant. /compliance/drafts/{id}/apply no-op except policy. /compliance-summary/generate generates nothing. SSO/OIDC /test are **offline field validators, not connectivity tests**. SCIM PATCH implements only op:replace on 3 paths, silently ignores everything else while returning 200. Dead: ScimTokenService.revoke_token, SCIMService._get_org_user, AttestationService.revoke_attestation, 3 unused ExportContentBuilder methods.

##### Other real issues
- **Exports have no async runner and no durable storage.** /run synchronous; binaries land in FILE_STORAGE_PATH (**default /tmp/complivibe_exports/**) so files vanish on restart while job still reads completed. No download endpoint. verify_job never re-reads the file → a swapped PDF verifies clean.
- Legal hold/retention are flag columns w/ one enforcement point (archive refusal); no purge/deletion anywhere.
- **Share-link passwords use unsalted hashlib.sha256** — should be bcrypt/argon2 (mitigated not fixed by 5-attempt lockout).
- **Permission mismatches**: POST evidence-manifests + verification export gated on org:read yet create rows AND auto-create signing keys. POST /reports/executive-narrative and /reports/regulatory/{type} write reports but require only reports:read. All four custom-role READ endpoints require org:update.
- **Cross-tenant weak spots**: SCIM provision_user looks up users globally by email before scoping to org → an IdP in org A can mutate a user row belonging to org B. run_daily_digest_dispatch sweeps EVERY org's subscriptions while permission-checked against caller's single org.
- **Role fallback ladders dangerous**: SCIM requesting "member" falls through reviewer→compliance_manager→admin→owner; if org lacks mid-tier roles, SCIM users land as **admin or owner**. Same in SSOService._resolve_default_role_id.
- **Import dry-run not enforced** — ImportJobService.commit() never reads job.dry_run; a dry_run=true job can be committed and writes real rows.
- Two parallel connector systems (cloud_evidence_connectors vs connector_catalog_entries/_org_enablements), no shared model. Two parallel LLM drafting stacks (ai_drafting = Azure-only direct; copilot_draft = Groq→Azure via AIProviderService). copilot_draft has no create endpoint → unusable standalone.
- **GET /health checks ONLY Postgres** — not Redis (now a rate-limiter dependency), R2/S3, LLM providers, or scheduler liveness; bypasses get_db so never exercises request-path pool.
- IP allowlist fails **open** when org has zero ranges (intended), derives client IP from first X-Forwarded-For hop w/ no trusted-proxy validation — spoofable unless proxy overwrites.
- Unauthenticated: /auth/register, /onboarding/start, /onboarding/accept-invite, /reports/shared/{token}, /roi-calculator, SCIM ServiceProviderConfig + Schemas. Only start/verify/roi carry rate limits — **accept-invite and GET /reports/shared/{token} have none**.
- PDF/DOCX renderers have hand-rolled fallbacks emitting near-empty but valid files on bare `except Exception` → any render error silently downgrades output rather than failing.

#### What overrides actually override
Break-glass exceptions to **export retention / legal-hold / attestation locks**. Four allow-listed requested_actions (governance_override_service.py:38-58): archive_locked_export, remove_legal_hold, adjust_retention_window (→export_job), revoke_attestation_after_lock (→export_attestation).

**Enforcement model is "executor", not "flag".** Nothing else in the codebase reads governance_override_requests (repo-wide grep hits only models/__init__.py, router.py, seed_service.py permission strings). The rows are effective because `execute()` (:1083) itself mutates the target:
- _execute_archive_locked_export (:973) sets export.status="archived" **directly**, deliberately NOT calling ExportService.archive_job (export_service.py:744-752) which would 400 on legal_hold or live locked_until. Re-checks legal_hold only (requiring a separate override to clear it first) but **does not re-check locked_until** — that is the point of the override.
- _execute_remove_legal_hold (:994) flips legal_hold=False, stamps reason w/ override id.
- _execute_adjust_retention_window (:1014) parses metadata_json and overwrites the columns.
- _execute_revoke_attestation_after_lock (:1058) calls AttestationService.revoke_attestation.
Every path writes an export_job_events row referencing override_request_id. **Not record-keeping — a real privileged-execution channel.**

#### Approval workflow
- **Justification**: reason mandatory on create/reject/cancel (stripped, empty→400). Optional on approve.
- **Quorum**: required_approvals >= 2 enforced at create (:722). Approvals increment approval_count; reaching quorum flips pending→approved. **One rejection immediately terminates → rejected.**
- **Separation of duties**: requester cannot approve or reject own request (:822, :893); duplicate decision by same approver blocked by lookup + IntegrityError→409 fallback behind SELECT...FOR UPDATE; if approver_role_names_json set, approver's active membership role must be in that list (403).
- **Expiry — LAZILY enforced only.** _ensure_pending_and_not_expired blocks approve/reject; execute() re-checks. Status only becomes `expired` when someone calls POST /governance/overrides/expire. **expire_pending has NO scheduler caller** — referenced only by that endpoint and its own definition.

#### Permission codes (7, seeded seed_service.py:205-211)
retention:read, retention:write, governance_override:create/read/approve/execute/cancel, governance_override_template:read/write. Owner/admin get all; reviewer/auditor-ish get read + governance_override:approve.

- **Two unrelated RetentionService classes**: app/services/retention_service.py (this feature, `retention_policies`) vs app/data_observability/services/retention_service.py (`data_retention_policies`). Same class name, different tables — easy to conflate.

## 7. Cross-cutting findings (rolled up from the domain walks)

These surfaced while reading code for this inventory. They are recorded here because they materially
change what a given endpoint count *means* — a feature that exists but never runs is not the same as a
feature that works. **None of these were fixed as part of this regeneration** (this commit is docs-only).

### 7.1 Highest severity

1. **Governance signing-key rotation/revocation is cosmetic.** `_derived_signing_key_material`
   (`app/api/v1/organizations.py:149`) derives `HMAC(SECRET_KEY, "{org_id}:{purpose}:{key_id}")` on
   demand — no key material is ever stored or destroyed, so a "revoked" key still produces valid
   signatures and verification only downgrades a `trusted` flag. Anyone with `SECRET_KEY` can forge any
   org's manifest. Note the platform now has **two signing models of very different strength**: this
   HMAC scheme, and P3's real Ed25519 (`policy_derivation/receipt_crypto.py`).
2. **Custom roles have no permission ceiling.** `_validate_permission_codes` only checks that a code
   exists. A holder of `org:update` can mint a role containing *any* permission and assign it to any
   membership — no check that the creator holds what they grant, no self-assignment block. Genuine
   privilege-escalation path.
3. **SSO/OIDC tokens drifted away from the cookie rearchitecture.** `SSOService._create_jwt_for_user`
   (`sso_service.py:215`) mints a token with no `jti` and no `csrf` claim, creates no `user_sessions`
   row, and the callbacks never set auth cookies. SSO-issued tokens are unrevocable, invisible to
   `/sessions`, and cannot perform mutating requests as a cookie — **SSO login does not produce a
   usable browser session in production**.
4. **Unauthenticated competitor-pricing exposure.** `trust_center_service.py:152` unconditionally
   returns the platform-global competitor pricing table (competitor names, pricing, source URLs) from
   `GET /trust-center/{slug}`, whose router has only `get_db` as a dependency — no auth, no `show_*`
   toggle, not org-scoped. *(Verified directly during this pass.)*
5. **Cross-tenant weak spot in SCIM.** `provision_user` looks up users globally by email before scoping
   to org, so an IdP in org A can mutate a user row belonging to org B. Separately, SCIM/SSO role
   fallback ladders (`member → reviewer → compliance_manager → admin → owner`) can land provisioned
   users as **admin or owner** if an org lacks the mid-tier roles.

### 7.4 Structural duplication

### 7.6 Configuration cliffs

- **Rate limiting is Redis-backed but opt-in.** `RATE_LIMIT_REDIS_URL` defaults to `None`; unset, it
  silently degrades to per-worker in-memory counting and the *N-workers = N×limit* cross-worker bypass
  returns. Set it in every multi-worker deployment.
- **IP allowlist fails open** when an org has zero ranges (intended), and derives client IP from the
  first `X-Forwarded-For` hop with no trusted-proxy validation.
- **Unauthenticated endpoints without rate limits:** `POST /onboarding/accept-invite` and
  `GET /reports/shared/{token}`. `/whistleblower/*` is not in the `public` limit group (falls through to
  `api_general` 300/min) and report bodies are stored unencrypted.
- **Import dry-run is not enforced** — `ImportJobService.commit()` never reads `job.dry_run`, so a
  `dry_run=true` job can be committed and write real rows.

### 7.7 Open question (not asserted as a bug)

`carbon_accounting_service.py:24` sets `UNIT_TO_TCO2E["MTCO2e"] = 1_000_000`. On the reading that
`MT` = metric tonne this would inflate readings 10⁶×; on the reading that `MT` = megatonne it is
correct. Since `tCO2e` already covers tonnes, a redundant third tonne-unit is unlikely and the
megatonne reading is self-consistent. **Flagged for owner confirmation, not recorded as a defect.**

## 8. How to regenerate this file

Do **not** hand-edit counts or append to the previous version — that is what produced the drift this
regeneration corrected. Re-derive everything:

```bash
# 1. Route count — from the live app, not from grep
.venv/bin/python -c "
from app.main import app
s = app.openapi()
ops = [(p,m) for p,ms in s['paths'].items() for m in ms
       if m in ('get','post','put','patch','delete')]
print('paths:', len(s['paths']), 'operations:', len(ops))"

# 2. Table count — from a real DB migrated to head, NOT from counting migration files
alembic upgrade head
psql -d <db> -tAc "SELECT count(*) FROM information_schema.tables
                   WHERE table_schema='public' AND table_type='BASE TABLE';"

# 3. Permission codes + per-role grants
.venv/bin/python -c "
from app.services.seed_service import PERMISSIONS, ROLE_PERMISSION_MAP
print(len(PERMISSIONS))
[print(r, len(p)) for r,p in ROLE_PERMISSION_MAP.items()]"

# 4. Router modules — routers are LAZILY included; call openapi() first, then walk
#    fastapi.routing._IncludedRouter via .original_router / .include_context.prefix.

# 5. Scheduled jobs
```

1. Per-domain endpoint counts sum to the live OpenAPI operation total.
2. Per-domain table counts sum to the live DB table total minus `alembic_version`.
3. Model-declared tables match DB tables exactly in both directions.
4. The §5 feature-row count, parsed structurally (>=4 cells, excluding `| feature |` headers and

   count rows with a slug regex — feature names carry parentheticals and dots, and a regex silently
   undercounts. This gate exists because it caught exactly that error (394 → 399).
